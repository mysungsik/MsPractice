[3. DBMS(DateBase Managament Systems) + 데이터베이스란?]
    : 단순한, 데이터베이스 처리를 위한 소프트웨어
    : 읽기 쓰기를 간단하게, 데이터 저장의 도움, data 쿼리의 도움 을 줌

    [데이터베이스]
        : 데이터의 저장 및 검색을 최적화하는 소프트웨어

    [웹사이트가 있다 == 데이터베이스가 있다.]
    
    [종류]
-       : SQL database(Structured Query Langauge)
            => 관계형 [RDBMS : relation DBMS]
            => [데이터베이스에, 데이터를 저장하기 위해], 구축된 [프로그래밍 언어]
            => [정규화된 데이터]를 여러 [테이블에 저장]

-       : NoSQL database
            => 비관계형
            => [데이터베이스에, 데이터를 저장하기 위해], 구축된 [프로그래밍 언어]
            => SQL을 사용하지 않고, 다른 Query 언어를 사용

[4. 쿼리, 스키마]
    : '질문' 으로 해석가능함. [어떤 데이터의 위치나 값을 가져오기 위한] '질문'
        쿼리를 한다 ==> 그에 맞는 데이터값을 가져온다.

    : 스키마란, 값에대해 명확하게 설명하는 텍스트

[5. SQL의 동작원리]
    : 수많은 테이블을 만들어, 테이블 안의 각 칸에 데이터를 저장, 위치를 기억, 위치의 데이터를 뽑고 사용
        => [정규화된 데이터]가 [여러 테이블에 저장]되어, 값을 사용하는 스타일

    1. SQL이 테이블을 생성, 데이터베이스는 [스키마와 데이터 타입]...
    2. [쿼리를 실행] [고유식별자를 통해 값의 위치를 찾고] [ 값을 얻음 ], [얻은 값을 고유식별자를 통해 다른곳에 집어넣음]
    3. [수많은 테이블이 존재] [서로 인용...]
    - ... 직접 하면서보자.

[6. NoSQL의 동작원리]
    : 문서 상에 데이터를 정리하여 사용 [객체나 파일로]
    
    1. 많은 양의 데이터가, 문서상에 정리되어, [SQL보다 한 파일에 많은 양의 데이터가 저장되므로,]
        [더 적은 쿼리로 더 많은 데이터를 얻을 수 잇음]

[7. SQL VS NpSQL]
    : 어떤것을 사용하고, 이유는 무엇인가

    [SQL]   1. 많은 구조와 규칙을 제공 [데이터 처리의 실수가 발생되기 힘듬 - 쿼리가 어려움]
            2. 스키마가 존재 [세세한 값의 적용이 가능]
            3. 적은 양의 데이터를 읽기 좋음 - 복잡한 쿼리 - 세세한 값

            => [스케일이 작은 곳 사용하기 좋음]

    [NoSQL] 1. 적은 구조와 규칙을 제공  [커다란 데이터 처리에 적합 - 쿼리가 쉬움]
            2. 스키마가 없음 [ 커다란 사례를 처리하기 좋음, 유연한 적용]
            3. 많은 양의 데이터를 읽기 좋음 - 간단한 쿼리 - 많양의 값

            => [스케일이 큰 곳 사용하기 좋음]
